## Context

The warm-up session currently shows keyword chips generated by a Gemini `generateText()` call on every config request. This adds ~1s latency, costs API quota, and produces inconsistent results (short fragments that don't help candidates form sentences). The 30 topics are already static in `constants/warmupTopics.ts`; adding phrases there is the natural extension.

Current data flow:
```
WarmupDashboard (topic selected)
  → POST /api/warmup/config?topic=...
    → generateKeywords(topic, level) [Gemini call]
    → returns { systemPrompt, keywords: string[], userLevel, streak }
  → WarmupSession renders keywords as horizontal pill chips
```

Target data flow:
```
WarmupDashboard (topic selected)
  → POST /api/warmup/config?topic=...&topicId=...
    → looks up static WARMUP_TOPIC_PHRASES[topicId]
    → returns { systemPrompt, phrases: string[], userLevel, streak }
  → WarmupSession renders phrases as vertical scrollable list
```

## Goals / Non-Goals

**Goals:**
- Eliminate the `generateKeywords()` Gemini call (remove latency + cost)
- Provide 5–7 actionable French sentence starters per topic, authored for TEF Canada B1–B2 level
- Phrases visible during the session as a light reading aid (not interactive)
- Easy tab for simpler sentence structures; medium tab for more complex constructions

**Non-Goals:**
- Personalizing phrases per user level (same phrases for all levels — level personalization is in the system prompt)
- Making phrases interactive (tap-to-speak, copy, etc.)
- Translating phrases to English

## Decisions

### 1. Static phrases in `constants/warmupTopics.ts` (not server-side)

**Decision:** Add `phrases: string[]` directly to each `WarmupTopic` object in the existing constant file.

**Rationale:** Topics are already static client-side constants. Co-locating phrases keeps the data together, avoids a new file, and means the frontend always has phrases available without a network round-trip. The server config endpoint simply returns `phrases` from this map (imported or duplicated server-side).

**Alternative considered:** A separate `WARMUP_PHRASES` map keyed by topic ID. Rejected — more indirection for no gain.

### 2. Server reads phrases from a static map, not from the client

**Decision:** Duplicate the `topicId → phrases` mapping server-side in `warmupService.ts` as a plain `Record<string, string[]>`. The config endpoint looks up phrases by `topicId` query param.

**Rationale:** Server is the source of truth for the config response. The client constant is the authoring source; the server map is a copy. Keeping them separate avoids bundling client constants into the server bundle (different module resolution paths).

**Alternative considered:** Have the client pass its own phrases up to the server and echo them back. Rejected — server should not trust arbitrary client-provided content for what it puts in responses.

### 3. Wire key stays `keywords` for now, renamed to `phrases` in types

**Decision:** Rename the TypeScript field to `phrases` in both frontend and backend types, and use `phrases` as the JSON key. The old `keywords` field is gone.

**Rationale:** Clean break — no backwards compat concern since this is a single-tenant app with no versioned API consumers.

### 4. Display: vertical scrollable list replacing horizontal chips

**Decision:** Replace the `flex flex-wrap gap-2` chip row with a compact vertical list inside a scrollable container (`max-h-36 overflow-y-auto`), each phrase on its own row with a subtle left border accent.

**Rationale:** Sentence starters are 5–10 words long — they don't fit in pill chips. A vertical list is scannable while the mic is active. Max-height + scroll keeps the UI from pushing the mic button off screen on small phones.

## Risks / Trade-offs

- **Content quality** — Static phrases authored once; if they're awkward or too formal, users are stuck with them until next deploy. Mitigation: write at TEF B1–B2 register, review before merge.
- **Topic drift** — If topics are later edited/added, phrases must be kept in sync manually. Mitigation: topics and phrases live in the same object in `warmupTopics.ts`, so a missing `phrases` field will be a TypeScript error.
- **Server-side duplication** — Phrases exist in both `constants/warmupTopics.ts` and a server-side map. Mitigation: document in a comment that the server map mirrors the client constant; a future refactor could share via a monorepo package.

## Migration Plan

1. Update `constants/warmupTopics.ts` (add `phrases` to interface + all 30 topics)
2. Add server-side `TOPIC_PHRASES` map in `warmupService.ts`, remove `generateKeywords()`
3. Update config route to use static phrases
4. Update frontend types + `WarmupView.tsx` plumbing
5. Update `WarmupSession.tsx` display
6. No data migration needed — session model doesn't store keywords/phrases
