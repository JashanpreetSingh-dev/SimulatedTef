## Problem Statement

D2C users practice oral and written expression, receive AI feedback identifying their
weaknesses (grammarNotes, vocabularyNotes, weaknesses[]), and then... repeat the same
mistakes next session. The feedback is ephemeral — read once, then forgotten. There is
no mechanism to close the loop between "AI identified a problem" and "user resolves it."

## Demand Evidence

- Founder is a daily user of the product and observes this pattern firsthand
- Pattern: practicing without targeted drills means the same errors recur session after session
- Status quo: no external study workflow — users just do more sessions and hope it sticks
- No direct user requests yet, but the product is early and founder-user insight is real demand

## Status Quo

After a session ends, users read their weaknesses/grammar notes, absorb them mentally,
and return for another practice session. No structured mechanism to target specific errors.
The Daily Ritual exists but is generic — not personalized to individual mistake history.

## Target User & Narrowest Wedge

**The buyer:** D2C user on Akseli ($45/mo premium). Skilled worker immigrant preparing for
TEF Canada. High stakes — a bad score delays PR application by 12-24 months. Has done
multiple practice sessions; is accumulating results.

**The wedge:** A single "Replay" button on the result detail view. Click it → get shown
your exact original utterance/written text that triggered a specific weakness → asked to
fix it → Gemini re-evaluates the fix and confirms whether the error is resolved.

## Constraints

- D2C only. B2B users are on an org-managed learning path.
- No new data collection required — transcript + written responses + grammarNotes + session
  IDs are already stored in the Result model.
- Must feel motivating, not punishing. Showing old mistakes requires careful UX framing.
- Gemini API calls must stay within cost envelope — pre-generation or caching needed.

## Premises

1. The Result model already has grammarNotes, vocabularyNotes, and weaknesses[] per session
   — no new data collection needed.
2. Users repeat the same French mistakes because there's no targeted drill mechanism after
   feedback.
3. The value is longitudinal — patterns emerge across 5-10 sessions, not one.
4. Gemini can generate targeted quiz questions from natural-language weakness descriptions.
5. D2C-only feature — B2B users have a different path managed by their org.

## Cross-Model Perspective

**SECOND OPINION (Claude subagent) — Key excerpts:**

"Instead of generating abstract quiz questions about 'subjunctive mood errors,' the system
retrieves the exact sentence the user produced that triggered the weakness flag, shows it
alongside their original AI feedback, and asks them to rewrite or re-speak it — then
re-evaluates with Gemini using the same rubric."

"The weakness is anchored to autobiographical memory, which dramatically increases retention.
The data is already there: session ID, utterance text, grammarNotes, weaknesses[]."

"Build the aggregation first. Validate that it surfaces real recurring patterns before
building any quiz UI. The quiz is only valuable if the input signal is good."

"The real value is the aggregation layer that answers: across all of this user's sessions,
what is the single most persistent gap?"

One challenged premise: subagent noted the normalization problem ("subjonctif incorrect"
vs "subjunctive errors" may need NLP/embeddings to deduplicate). Not a blocker for MVP
but a known risk for the aggregation layer.

## Approaches Considered

### Approach A: Post-Session Drill (Minimal Viable)
Generate 3-5 MCQs from the current session's weaknesses immediately after results.
Effort: S. Risk: Low. Pros: ships fast, validates Gemini quality. Cons: no longitudinal
value, ephemeral.

### Approach B: Weakness Dashboard + Drill Bank (Ideal Architecture)
MongoDB aggregation across all user results surfaces top recurring weaknesses.
Dedicated /weaknesses page with frequency counts + on-demand drill + ts-fsrs scheduling.
Effort: M. Risk: Medium (aggregation normalization is the hard part).

### Approach C: Mistake Replay Mode (CHOSEN)
Skip abstract MCQs. Surface the user's exact past utterances/written text that triggered
weakness flags. Ask them to rewrite or re-speak. Gemini re-evaluates. Loop closes.
Effort: M (1 week). Risk: Medium-Low.

## Recommended Approach: C — Mistake Replay Mode

**Why this over A or B:**

Approach A validates Gemini quality but never gives users pattern visibility. Approach B
has the right ambition but the cold start problem (needs 5-10 sessions of history) and
aggregation normalization complexity make it a second-month feature.

Approach C is simultaneously simpler than B AND more differentiated than either:
- No question generation quality problem — you're not writing abstract MCQs, you're
  surfacing the user's own words back to them
- Anchored to autobiographical memory — stronger retention than generic flashcards
- The data is already there — transcript, grammarNotes, Result IDs
- Nobody else does this. Duolingo tracks wrong answers and moves on. Akseli shows you
  the exact moment you failed and makes you resolve it.

**Implementation sketch:**

**Prerequisite:** Verify written text storage before writing any code. Run:
`db.results.findOne({ module: 'writtenExpression', 'moduleData.sectionA.text': { $exists: true } })`
in production MongoDB. If `sectionA.text` is absent on older records, the Drill button
must be hidden for those results (no fallback text available).

**Scope: MVP uses text input for all module types.** Oral mic input requires the full
recording pipeline (S3 upload, BullMQ eval job) — too heavy for the first ship. Both
oral and written Replay use a textarea for corrections in MVP. Mic-based oral Replay
is Phase 1.5.

**Excluded from Replay:** Reading and Listening module results. Those are MCQ-scored — no
free-text utterance to replay. Drill button must not appear for those result types.

1. **Replay surface:** Add a "Drill this weakness" button to each item in the weaknesses
   list inside `DetailedResultView` (rendered via `StrengthsWeaknesses.tsx`). One button
   per weakness string. No session-level "Replay" CTA — per-weakness is more focused.
   Button is hidden if: (a) module is reading/listening, (b) written text / transcript is
   absent, (c) transcript is present but fewer than 30 characters.

2. **Backend: `POST /api/drill/replay`** (protected by `requireAuth`)
   - Accepts: `{ resultId, weakness: string, section?: 'A' | 'B' }`
   - Use `weakness` (the literal string) as identifier — NOT an array index. Array position
     is fragile if weaknesses[] is ever re-evaluated or re-ordered.
   - `section` is required for writtenExpression results. Backend field-access map:
     - writtenExpression + section A → `moduleData.sectionA.text`, `moduleData.sectionA.weaknesses[]`
     - writtenExpression + section B → `moduleData.sectionB.text`, `moduleData.sectionB.weaknesses[]`
     - oral → `transcript` (top-level field on Result), `weaknesses[]` (top-level)
     - If `section` is present on an oral result: reject with 400.
   - Oral transcript field path: `result.transcript` (top-level string). Prerequisite check:
     `db.results.findOne({ module: 'oral', transcript: { $exists: true, $ne: '' } })`
   - Returns: `{ originalText, weaknessDescription, drillPrompt, sessionType }`
   - Gemini error handling: if Gemini returns malformed JSON or times out, retry once.
     On second failure: return `{ error: 'drill_unavailable' }` — frontend shows
     "This drill isn't available right now. Try again later."
   - Cache key: `drill:replay:{resultId}:{weaknessHash}` where weaknessHash = first 8 chars
     of sha256(weakness). Redis, 24h TTL.
     If cache hit: return cached payload, skip MongoDB write.
     If cache miss: call Gemini, write to DrillReplay collection, cache result.

3. **Gemini prompt A — drill generation:**
   ```
   You are a TEF Canada French examiner coaching a student.
   The student wrote/said this: "{originalText}"
   The AI examiner identified this weakness: "{weakness}"
   Your job: show the student exactly what was wrong in their text, then ask them to
   rewrite the problematic section only. Keep it focused.
   Return JSON: { problematicSection: string, explanation: string, rewritePrompt: string }
   ```
   `problematicSection` is a substring of `originalText`. Highlighting: wrap in a
   `<mark>` tag using `indexOf(problematicSection)` to find position. If the substring
   appears multiple times, fall back to displaying the full text without highlight
   (no mark tag).

4. **Gemini prompt B — fix evaluation:** (called on submit, always live, ~3s budget)
   ```
   You are a TEF Canada French examiner. The student made this error: "{weakness}"
   The problematic text was: "{problematicSection}"
   The student's correction is: "{userCorrection}"
   Does the correction resolve the identified weakness? Be strict.
   Return JSON: { corrected: boolean, explanation: string }
   ```
   Same error handling as prompt A: retry once on malformed JSON or timeout, then 500.
   Streaming is available in the existing Gemini client — use it to reduce perceived
   latency on mobile. Stream the `explanation` field character by character; `corrected`
   boolean is parsed from the complete response.

5. **Replay UI (`/drill/replay/:resultId?weaknessIndex=N`):**
   - Shows: original text with `problematicSection` highlighted (or full text if ambiguous)
   - Shows: one-line explanation of what's wrong
   - Shows: rewrite prompt ("Rewrite this sentence using the correct subjunctive form")
   - Input: textarea for the user's written correction (all module types in MVP)
   - On submit: calls Gemini with prompt B, records attempt in DrillReplay.attempts[]
   - Result: "Fixed!" or "Still needs work — try again"
   - "Try again" re-uses the same cached drillPrompt from step 3 (no re-generation).
     The evaluation call (prompt B) is always fresh — not cached.
   - If `corrected: true`: set `resolvedAt = now()` on the DrillReplay document.
   - Frontend route passes `resultId` from URL path param in the POST body.

6. **`DrillReplay` MongoDB collection schema:**
   ```typescript
   {
     userId: string,
     resultId: string,
     weakness: string,         // literal string, not index
     section?: 'A' | 'B',
     resolvedAt?: Date,        // set on first corrected: true verdict
     attempts: Array<{
       attemptedAt: Date,
       userResponse: string,
       verdict: 'resolved' | 'still_wrong',
       geminiComment?: string,
     }>
   }
   ```

7. **Phase 2 (separate design, after MVP validation):** Weakness aggregation across
   sessions, recurring pattern detection, /weaknesses dashboard page. Spaced repetition
   scheduling (ts-fsrs) and dashboard badge deferred to Phase 2 design. Phase 2 begins
   only after MVP Replay is validated — if aggregation surfaces noisy or irrelevant
   weakness clusters, NLP normalization (embeddings) may be required.

**Data already available:**
- `transcript` — oral session text, stored in Result
- Written responses — `moduleData.sectionA.text` / `sectionB.text` on writtenExpression results
- `grammarNotes` (string), `vocabularyNotes` (string) — free-text per session
- `weaknesses[]` (string[]) — structured array per EvaluationResult, used for Replay
- `Result._id` — enables lookup of specific session data

## Open Questions

1. **Written text field path (blocking):** Run the MongoDB query above before starting.
   If `moduleData.sectionA.text` is absent in production records, written Replay is
   blocked until that field is added to the evaluation save path.
2. **Oral transcript quality:** How accurate is the speech-to-text transcript for short
   utterances? If transcripts are frequently < 30 chars or garbled, oral Replay (even
   text-only in MVP) may need a minimum quality threshold before showing the Drill button.
3. **Mobile UX:** Showing highlighted text + rewrite prompt on a small screen — needs
   manual testing before ship. The GuidedWritingEditor textarea is a reference for
   mobile-friendly French text input.
4. **Gemini response time:** Each Replay = 2 Gemini calls (generate + evaluate). Redis
   caching (see implementation) handles the first call. The second call (on submit) is
   always live — budget ~3s for the user. Show a loading indicator.

## Success Criteria

- Users who engage with Mistake Replay at least once show lower recurrence of the drilled
  weakness in subsequent sessions (measurable via weakness[] arrays across sessions)
- Replay feature is used by >30% of D2C premium users within 30 days of launch
- Gemini fix evaluation returns `corrected: true` for at least 50% of replay attempts
  (proxy for question quality and user engagement)
