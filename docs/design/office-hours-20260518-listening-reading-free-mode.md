# Design: Simplified Practice — Free Reading/Listening + Remove Mock Exam

Generated by /office-hours on 2026-05-18
Branch: listening-reading-free-mode
Repo: tef-canada-master
Status: DRAFT
Mode: Startup

## Problem Statement

Akseli currently has 5 practice entry points for D2C users: Oral Expression, Written
Expression, Reading Comprehension (hidden), Listening Comprehension (hidden), and a
separate Mock Exam tab. The mock exam has low engagement and was built because it felt
expected, not because users demanded it. Reading and Listening are completely hidden from
D2C users — they're wired to a B2B assignment workflow. The product is more complex than
it needs to be for the value it delivers.

## Demand Evidence

- Mock exam: low usage across the user base. No user has expressed panic about losing it.
- Oral/written AI feedback: the actual conversion trigger. Users hit the AI grading
  experience, get hooked, hit the usage limit, and subscribe.
- Reading/listening: currently invisible to D2C users — zero data on D2C demand, but
  zero marginal cost to serve (auto-graded MCQ, no AI call required).

## Status Quo

D2C users open Akseli and see Practice → 2 modules (oral, written). They do oral +
written sessions, rarely venture to the mock exam tab. The mock exam tab exists in nav
but goes largely unvisited. Reading and listening are inaccessible to them.

## Target User & Narrowest Wedge

**User:** D2C subscriber preparing for TEF Canada. Wants to practice consistently,
prefers short focused sessions over full timed exams.

**Wedge:** Make reading/listening available as free, unlimited, self-service practice
tabs alongside oral and written. Remove the mock exam to simplify navigation. The bet:
free reading/listening → habit-building → users discover oral/written AI feedback →
subscribe.

## Constraints

- `ReadingComprehensionExam.tsx` and `ListeningComprehensionExam.tsx` already exist
  as standalone components and support non-assignment mode. The work is wiring them
  into a new D2C entry point, not building new exam components.
- Current `ReadingTab` and `ListeningTab` in practice are B2B assignment flows. New
  D2C-specific tab components are required.
- No usage limit on reading/listening (auto-graded, no AI cost).
- Results must be saved to history so users can track progress.
- Mock exam removal touches more than just the route — see Step 5 cleanup list.

## Premises

1. Mock exams add navigational complexity with no measurable return — low usage, no
   user demand evidence, built from assumption not validation.
2. Reading + listening are auto-scored MCQ — zero AI cost per session, safe to offer
   free without margin impact.
3. The conversion flywheel: free reading/listening → habit → user wants AI-graded
   oral/written → subscribes. Structurally sound given oral/written AI is the moat.
4. The work is extraction + backend change + D2C wiring (~3-4 days), not a ground-up
   build. The core exam components already exist.

## Approaches Considered

### Approach A: Full Simplification (CHOSEN)
Remove mock exam tab + route entirely. Extract reading/listening from mock exam into
standalone D2C practice modules. Unlimited retries, history saved. All 4 sections in
one Practice page.
- Effort: M (~3-4 days)
- Risk: Low
- Pros: clean UX, free to serve, flywheel hypothesis, less nav complexity
- Cons: small % mock exam users lose that experience (low risk given low usage)

### Approach B: Keep Mock Exam + Add Free Practice
Keep mock exam, also add reading/listening to practice as free sections. Two entry
points to the same exams.
- Pros: nobody loses anything
- Cons: user confusion ("what's the difference?"), adds complexity, doesn't cut debt

### Approach C: Premium Simulation Mode
Reframe mock exam as a paid premium "simulation day" feature.
- Pros: potential tier differentiator
- Cons: building paywall around a feature nobody uses; backwards prioritization

## Recommended Approach

**Approach A — Full Simplification.**

The data says cut it. Low-usage features that weren't validated shouldn't survive.
Reading/listening free mode is the right experiment: it costs nothing to serve,
it puts more value in front of users before they subscribe, and it doesn't dilute
the oral/written AI moat (which is what people actually pay for).

### Implementation plan

**Pre-condition: confirm B2B reading/listening behavior**

Before starting: decide whether B2B users (professors/students) should also see the
new free-mode D2C reading/listening tabs in Practice, or keep B2B practice
reading/listening assignment-only. This decision affects Step 4. Both are valid — but
pick one before implementation starts. Recommendation: keep B2B assignment-only for
now (less regression risk), add free-mode for D2C only.

**Step 1 — Backend: make `mockExamId` optional, fix `resultType` for free-mode**

This is the load-bearing backend change. Without it, nothing else works.
Changes required across three layers (all in the same PR):

- `server/routes/exam.ts:445`: make `mockExamId` optional in route validation. When
  absent → free-mode path. When present → existing mock exam path.
- `server/controllers/mcqController.ts:26`: change `mockExamId: string` →
  `mockExamId?: string`. Internally, set `result.mockExamId` only when provided (~line
  104). Change hardcoded `title: 'Mock Exam - Reading/Listening'` (~line 105) to use
  the task's own title when `mockExamId` is absent.
- **Critical — `resultType` for history (three files must change together):**
  `mcqController` hardcodes `resultType: 'mockExam'` (~line 101). `HistoryList.tsx:49-52`
  queries `resultType: 'assignment'` for reading/listening modules. Free-mode results
  saved as `'mockExam'` will never show in History.
  Fix: use `resultType: 'practice'` for free-mode submissions (consistent with how oral/
  written use `'practice'`).
  - `server/services/resultsService.ts:24`: add `'practice'` to the `resultType` union
    if not already present (oral/written likely already use it — confirm).
  - `server/controllers/resultsController.ts:28`: update the query cast to include
    `'practice'` for the reading/listening history query.
  - `components/HistoryList.tsx:49-52`: update the query for reading/listening module
    history to include `resultType: 'practice'` alongside `'assignment'`.
- **Session ID:** For free-mode, the frontend should generate a `sessionId`
  (`crypto.randomUUID()` or similar) at the time the user starts an exercise, the
  same way mock exam generates one. Add this to the D2C tab component (Step 3).
- **Rate limiter:** `mcqSubmissionLimiter` (5 req/min, `server/middleware/rateLimiter.ts:52`)
  will hit on rapid retries. Mitigation: on 429, show a user-readable "Try again in Xs"
  message in the exam component rather than a generic error. Exempting the path entirely
  is also valid but requires a separate decision.

**Step 2 — Frontend exam component changes**

Changes to shared components — must be regression-tested against B2B paths.

- `ReadingComprehensionExam.tsx:14`: change `mockExamId: string` → `mockExamId?: string`.
  Same for `ListeningComprehensionExam.tsx`.
- **Timer fix (two guards, both must change):**
  1. `ReadingComprehensionExam.tsx:55` — initial state sets `TIME_LIMIT_SECONDS` when
     `assignmentId` is absent. Change to: set `Infinity` when `!assignmentId && !mockExamId`.
  2. `ReadingComprehensionExam.tsx:120` — the `setInterval` loop guard `if (!assignmentId)`
     starts the countdown. Change to: `if (!assignmentId && mockExamId)` so free-mode
     sessions (no assignmentId, no mockExamId) do not start the timer.
  Apply the same two-guard fix to `ListeningComprehensionExam.tsx`.
- **Rate limiter:** `mcqSubmissionLimiter` (5 req/min) will block rapid retries and
  contradict the "unlimited retries" promise. For free-mode: either exempt the
  free-mode path from this limiter, or surface a clear "slow down" message with a
  retry countdown rather than a generic 429 error.
- `limitReachedState.switchTo` type in `PracticeView.tsx:45` is `'oral' | 'written'`.
  Expand to `'oral' | 'written' | 'reading' | 'listening'` so deep-linking to a module
  via navigation state works for the new modules.

**Step 3 — New D2C reading/listening practice components**

- Create `components/practice/D2CReadingTab.tsx`:
  - Fetch available reading tasks via HTTP: `GET /api/reading-tasks/active` (or
    equivalent — check `server/routes/` for an existing route before creating one;
    `server/services/readingTaskService.ts:78` has `getAllActiveTasks()` server-side
    but there must be an HTTP route that exposes it to the frontend).
  - Generate a `sessionId = crypto.randomUUID()` when the user selects a task to
    start. Pass it to `<ReadingComprehensionExam>`.
  - Show a list/grid of tasks. User picks one, attempts it. Unlimited retries.
  - Render `<ReadingComprehensionExam taskId={task.id} sessionId={sessionId}>` with
    no `assignmentId` and no `mockExamId`.
  - **localStorage cleanup:** `ReadingComprehensionExam` saves progress under
    `reading_exam_{taskId}_{sessionId}`. In free-mode, clear this key on completion
    so the next attempt starts fresh. Pass an `onComplete` callback that calls
    `localStorage.removeItem(storageKey)`.
  - **Empty state:** if the API returns zero tasks, show "No exercises available yet"
    with a friendly message. Do not render a broken exam selector.
- Create `components/practice/D2CListeningTab.tsx` — same pattern for listening.
  Same `sessionId` generation, same localStorage cleanup, same empty state handling.
- Model component structure after `ExpressionOraleTab.tsx` for UX consistency
  (exam card chooser, back-navigation pattern, session state).

**Step 4 — Add reading/listening to PracticeView D2C branch**

The D2C path in `PracticeView.tsx` renders an inline two-tab switcher (oral, written)
at line ~156. Add reading and listening to it. **Do not collapse the D2C/B2B branch
structure** — that refactor is out of scope and increases regression risk in B2B paths.
Add the two new modules to the existing D2C inline tab switcher.

Minimal change set:
- `PracticeView.tsx:45`: expand `limitReachedState.switchTo` to
  `'oral' | 'written' | 'reading' | 'listening'`.
- `PracticeView.tsx` D2C branch: add reading and listening cards to the inline module
  selector (currently only shows oral + written). Wire:
  `selectedModule === 'reading'` → `<D2CReadingTab>`,
  `selectedModule === 'listening'` → `<D2CListeningTab>`.
- **Note:** `PracticeTabNavigation.tsx:7` already has the correct type
  `'oral' | 'written' | 'reading' | 'listening'`. Remove the `as 'oral' | 'written'`
  TypeScript cast in `PracticeView.tsx:199` that was narrowing it.
- `PracticeModuleSelector.tsx:54,70`: these `!isD2C` guards are logically redundant
  (D2C users don't reach `PracticeModuleSelector` — the parent renders the D2C inline
  switcher instead). Leave them or remove them — either is correct. Do not treat this
  as a required change.

**Optional follow-up (separate PR):** unify D2C and B2B into a single
`PracticeModuleSelector` code path to remove the duplication. Not required for this feature.

**Step 5 — History wiring**

- `PracticeView.tsx:212` D2C branch: currently hard-codes
  `module={selectedModule === 'oral' ? 'oralExpression' : 'writtenExpression'}`.
  This must be extended: add reading and listening cases:
  ```
  selectedModule === 'reading' ? 'readingComprehension' :
  selectedModule === 'listening' ? 'listeningComprehension' :
  selectedModule === 'oral' ? 'oralExpression' : 'writtenExpression'
  ```
- `HistoryList.tsx:49-52`: update the query for `readingComprehension` and
  `listeningComprehension` modules to include `resultType: 'practice'` (consistent
  with Step 1's decision; do NOT use `'freeMode'` — that value does not exist in the
  codebase).
- **Retake flow for free-mode results:** `HistoryList.tsx:517-538` branches on
  `assignmentId` for the Retake button. For free-mode results (no `assignmentId`, no
  `mockExamId`), the code falls through to the mock exam branch and shows
  `alert('Unable to retake this exam. Mock exam information is missing.')` — which is
  wrong. Add a free-mode branch: if no `assignmentId` and no `mockExamId`, navigate
  to `/practice` (let the user start a fresh session from the Practice tab).
  The `taskId` from the result could optionally be passed as state to pre-select the
  same task, but the simple fallback of `/practice` is acceptable for v1.

## Reviewer Concerns

These items were flagged across multiple review rounds and were not fully resolved
in the design doc. An engineer should address them during implementation:

1. **Free-mode HTTP endpoint unclear:** Step 3 references `readingTaskService.getAllActiveTasks()`
   (a server-side method). Confirm there is a corresponding `GET /api/reading-tasks/active`
   (or equivalent) HTTP route. If not, create one. The document does not spec the response
   shape — use the same shape as the mock exam reading task fetch.

2. **`mcqController` question count validation:** `mcqController.ts:35` may validate for
   40 questions. If a free-mode reading/listening task has fewer questions, the controller
   will throw before saving. Verify and update the validation to accept variable-length
   answers.

3. **Timer guard line 55 exact condition:** The correct fix for the initializer is:
   `(!assignmentId && !mockExamId) ? Infinity : TIME_LIMIT_SECONDS`.
   The interval guard at line 120 should be `if (!assignmentId && mockExamId)`.
   Apply the same fixes to `ListeningComprehensionExam.tsx` — verify it has the same
   structure before assuming symmetry.

4. **Keyboard navigation in D2C tab switcher:** The current inline D2C switcher uses
   `ArrowRight`/`ArrowLeft` for two tabs. Adding reading/listening breaks the keyboard
   nav. Update the keyboard handlers when extending the switcher to 4 modules.
