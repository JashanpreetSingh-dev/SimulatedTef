# Plan: Next.js Conversion + Vercel Deployment?

Generated by /autoplan on 2026-04-03
Branch: master

## Question
Should we convert from the current Vite SPA + Railway deployment to Next.js + Vercel?

## Current Stack
- **Frontend:** React 19 + Vite (SPA), React Router v7
- **Backend:** Express.js + TypeScript (`server/server.ts`, port 3001)
- **Workers:** BullMQ workers — evaluationWorker, questionGenerationWorker, emailWorker
- **Queue/Cache:** Redis (BullMQ)
- **Database:** MongoDB
- **Real-time:** Gemini Live WebSocket PCM audio (`OralExpressionLive.tsx`)
- **Auth:** Clerk
- **Payments:** Stripe (webhooks at `/api/stripe-webhooks`)
- **Deployment:** Railway (single platform, frontend + backend + workers + Redis)

## What the conversion would involve
- Rewrite frontend from Vite SPA to Next.js App Router or Pages Router
- React Router v7 → Next.js file-based routing
- All Tailwind config (currently CDN-based in index.html)
- Clerk: `@clerk/clerk-react` → `@clerk/nextjs`
- Move some Express endpoints to Next.js API routes
- BUT: BullMQ workers, Redis, MongoDB, WebSockets cannot run on Vercel
  → Backend would stay on Railway anyway (or move to separate service)

### REVIEW SECTIONS

#### Section 1: Architecture Review

Current architecture (Railway monolith):
```
  Browser → Vite SPA → Express API (Railway :3001)
                      ↓
              MongoDB + Redis + BullMQ workers
                      ↓
              Gemini Live WebSocket (persistent, PCM)
              Stripe webhooks (raw body, single endpoint)
              Clerk webhooks (single endpoint)
```

Proposed Next.js/Vercel architecture:
```
  Browser → Next.js (Vercel)  →  Express API (Railway :3001)
                  ↓                      ↓
          Some API routes         MongoDB + Redis + BullMQ workers
          (ambiguous ownership)   Gemini Live WebSocket
                                  Stripe webhooks (WHERE?)
                                  Clerk webhooks (WHERE?)
```

**CRITICAL GAP:** Stripe webhook endpoint requires raw body access. Next.js API routes handle this differently than Express. The migration plan doesn't address webhook ownership. Stripe must be configured with a single endpoint URL — post-migration you have two candidate URLs.

**CRITICAL GAP:** Gemini Live PCM audio streams via persistent WebSocket. Vercel serverless functions have a 10-second (Hobby) to 300-second (Pro) timeout. Real-time oral practice sessions run 2-10 minutes. This **cannot** work on Vercel — it must stay on Railway. The migration creates a situation where the product's most important feature is explicitly excluded from the "new" architecture.

**COUPLING:** After migration, every Vercel deployment must be coordinated with Railway. CORS headers must be maintained between two domains. Clerk auth sessions must work cross-origin. This is strictly more coupling than the current single-domain deployment.

#### Section 2: Error & Rescue Map (Migration Risks)

| Migration Step | What Can Go Wrong | Severity |
|---|---|---|
| Clerk package swap | `@clerk/nextjs` middleware breaks auth silently | CRITICAL |
| Gemini Live + App Router | Missing `use client` → SSR crash in production | CRITICAL |
| Stripe webhooks | Raw body parsing fails in Next.js API route | CRITICAL |
| CORS | Vercel frontend can't reach Railway backend | HIGH |
| Tailwind CDN → build | CSS purging removes classes used dynamically | HIGH |
| React Router → file-based | `navigate()` patterns need rewrite | MEDIUM |
| BullMQ job results → frontend | Polling/WS still hits Railway; Vercel adds no value | MEDIUM |

#### Section 3: Security & Threat Model

Splitting to Vercel + Railway expands attack surface:
- Two CORS origins to maintain correctly
- Stripe webhook secret must be accessible in both environments or webhook moved to one — ambiguity creates misconfiguration risk
- Clerk JWT validation in Next.js middleware must handle cross-domain sessions correctly
- Any Railway API endpoint exposed to Vercel frontend is now a cross-origin API — all existing CORS assumptions must be audited

No net security benefit from migration.

#### Sections 4-10: Performance, Quality, Tests, Observability, Deploy, Trajectory

**Performance:** Vercel CDN for static assets is a marginal improvement. Vite build output is already optimally cacheable. Real compute (AI eval, audio, workers) stays on Railway. Net performance improvement: ~0.

**Code Quality:** Migration would temporarily degrade quality — ~150 files touched for routing conversion, auth changes, client boundary annotations. No new features shipped.

**Tests:** No existing E2E test suite. Migration would be unverified. BullMQ + Gemini Live flows particularly fragile without integration tests.

**Observability:** Railway currently has unified logs. Post-migration: split logs across two platforms. Harder to trace a user session that touches both Vercel API routes and Railway workers.

**Deployment:** Currently one `git push` deploys everything via Railway. Post-migration: two deploy targets, two environments to keep in sync, Railway + Vercel rollout sequencing required.

**Trajectory (10: Long-Term):** Migration does not move toward 12-month ideal. Creates operational debt. Reversibility: 2/5 (would need to migrate back). Tech debt introduced: ~2-3 weeks of cleanup per year maintaining two-platform coordination.
